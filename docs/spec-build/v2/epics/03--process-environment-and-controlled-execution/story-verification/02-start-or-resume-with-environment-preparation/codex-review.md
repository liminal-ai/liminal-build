CORRECTNESS_FINDINGS
- [High] AC-2.2 is not implemented for current outputs. Story 2 requires environment preparation to hydrate the process's current artifacts, current outputs, and current sources into the working copy (`stories/02-start-or-resume-with-environment-preparation.md:58-67`). The new working-set types only carry `artifactIds` and `sourceAttachmentIds` (`apps/platform/server/services/projects/platform-store.ts:63-71`), and `planHydrationWorkingSet()` copies only those two arrays (`apps/platform/server/services/processes/environment/hydration-planner.ts:1-13`). The new Story 2 action tests only assert those two fields as well (`tests/service/server/process-actions-api.test.ts:956-1042`). Current outputs never enter the preparation path, so the implementation cannot satisfy AC-2.2 as written.
- [High] AC-2.3 is still short in the current tree. The design for this slice expects `start` / `resume` to hand off to an environment service that resolves a provider, ensures an environment, hydrates it, and then publishes later `ready` or `failed` updates (`docs/spec-build/v2/epics/03--process-environment-and-controlled-execution/tech-design-server.md:786-798`). In the implementation, the `environment/` directory only contains `hydration-planner.ts`, `createApp()` wires no environment service or provider adapter layer at all (`apps/platform/server/app.ts:21-29`, `apps/platform/server/app.ts:110-121`), and `ProcessStartService` / `ProcessResumeService` only persist `environment.state = preparing`, optionally store a plan, and publish one immediate upsert (`apps/platform/server/services/processes/process-start.service.ts:33-76`, `apps/platform/server/services/processes/process-resume.service.ts:33-78`). There is no production path that can surface hydration progress or hydration failure after action acceptance without synthetic fixtures or manual hub publication.
- [High] AC-2.4 is violated because the accepted `start` / `resume` path still moves the process itself to `running` before readiness is confirmed. The underlying Convex mutations immediately call `transitionProcessToRunning()` for both actions (`convex/processes.ts:334-349`, `convex/processes.ts:702-726`), and the page renders `process.status` directly in the header (`apps/platform/client/features/processes/process-work-surface-page.ts:101-105`). The new tests explicitly codify this by expecting `Running` while the environment panel still says `Preparing environment` (`tests/service/server/process-actions-api.test.ts:806-891`, `tests/service/client/process-work-surface-page.test.ts:1343-1396`). That contradicts Story 2's requirement that active running work begins only after readiness and that the surface does not falsely present the process as actively running after failed preparation (`stories/02-start-or-resume-with-environment-preparation.md:80-89`).

ARCHITECTURE_FINDINGS
- The Story 2 implementation does not align with the chunk-2 server architecture. The design says `start` / `resume` should become thin wrappers over `ProcessEnvironmentService`, which then owns hydration planning, provider selection, and later lifecycle publication (`docs/spec-build/v2/epics/03--process-environment-and-controlled-execution/tech-design-server.md:786-806`). The current tree skips that layer entirely: app assembly wires only `ProcessStartService` / `ProcessResumeService`, and the `services/processes/environment/` directory contains only the planner helper (`apps/platform/server/app.ts:110-121`, `apps/platform/server/services/processes/environment/hydration-planner.ts:1-13`). This is not just an internal refactor gap; it is the reason AC-2.3 and AC-2.4 are currently unmet.
- The verification bundle's claim that hydration-plan persistence is wired through all store implementations is inaccurate for the production store (`story-verification/.../verification-bundle.md:37-45`). `ConvexPlatformStore.getProcessHydrationPlan()` always returns `null`, and `setProcessHydrationPlan()` simply echoes the input plan without persisting it (`apps/platform/server/services/projects/platform-store.ts:721-729`). Only the in-memory test store actually stores the plan. That makes the new planning layer test-only behavior today, not a real durable capability.
